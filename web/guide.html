How to Use & Interpret Results

A complete guide to testing file upload security

Getting Started

This tool helps you identify security vulnerabilities in file upload functionality. It generates 31 different test files including:

  • Corrupted images - Files with broken headers or wrong MIME types
  • Malicious documents - PDFs with embedded scripts, SVG with XSS
  • Archive bombs - Highly compressed files designed to exhaust resources
  • Path traversal - Files designed to escape upload directories
  • Double extensions - files.txt.exe that bypass simple filters
Legal Notice: Only test servers you own or have explicit written permission to test. Unauthorized access is illegal.

Using the CLI

For more advanced testing, use the command line interface:

# Basic test python3 -m cli.main run https://example.com/upload # With authentication python3 -m cli.main run https://example.com/upload --token "your-token" # Through proxy (Burp/ZAP) python3 -m cli.main run https://example.com/upload --proxy http://localhost:8080 # Concurrent testing (faster) python3 -m cli.main run https://example.com/upload --workers 10

Understanding Status Codes

When you upload a file, the server responds with an HTTP status code. Here's what they mean:

Success (2xx) - Usually Good

CodeMeaningSecurity Note
200 OKFile acceptedCheck if it's safe to accept this file type
201 CreatedFile savedVerify the file type is allowed
415 UnsupportedWrong file typeGood! Server validates MIME type

Client Errors (4xx) - Often Good

CodeMeaningSecurity Note
400 Bad RequestServer didn't understandMay indicate missing validation
403 ForbiddenAccess deniedGood! Server blocked the file
413 Payload Too LargeFile too bigGood! Server validates size

Server Errors (5xx) - Usually Bad

CodeMeaningSecurity Note
500 Internal ErrorServer crashedDanger! File causes crash - potential exploit
502 Bad GatewayServer errorMay indicate processing issues

What to Look For

High Acceptance Rate

If the server accepts more than 50% of test files, it likely lacks proper validation.

Server Crashes (5xx)

Critical! If any file causes a 500 error, the server may be vulnerable to denial of service or code execution attacks.

Error Messages in Response

If the response contains "SQL syntax", "Stack trace", or internal paths, the server may be leaking sensitive information.

Security Checklist

Use this checklist when reviewing file upload functionality:

  • Server only accepts documented/expected file types
  • MIME type is validated on server-side (not just client-side)
  • File content is inspected, not just extension
  • Maximum file size is enforced
  • Corrupted/malformed files are rejected
  • Path traversal (../) in filenames is blocked
  • Double extensions (file.txt.exe) are blocked
  • Null byte injection is handled
  • ZIP archives are validated for path traversal
  • Files with scripts (SVG, PDF with JS) are blocked
  • Upload attempts are logged for security monitoring
  • Internal error messages are not exposed to users

Interpreting Results

Good Results

  • Most files get 403, 415, or 413 responses
  • No 500 errors (server doesn't crash)
  • Clean error messages without internal details
  • Only expected file types are accepted

Bad Results

  • Server accepts 200+ for most test files
  • Any 500 errors on test files
  • SQL injection or stack traces in responses
  • Files saved with original names (path traversal possible)
  • No file type validation visible
